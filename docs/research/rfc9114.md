# RFC 9114 — HTTP/3

## Summary

RFC 9114 specifies HTTP/3: the mapping of HTTP request/response semantics
onto QUIC streams. Framing is simpler than HTTP/2 because QUIC provides
stream multiplexing, per-stream flow control, and head-of-line-blocking
avoidance, so HTTP/3 keeps only seven request/control frames (DATA,
HEADERS, CANCEL_PUSH, SETTINGS, PUSH_PROMISE, GOAWAY, MAX_PUSH_ID) plus
the "unknown frame MUST be ignored" rule from §7.2.8. In ExpressGateway
the `lb-h3` crate implements the frame codec (`decode_frame`,
`encode_frame`), QUIC varint (RFC 9000 §16) codec, and QPACK
decompression (covered separately in `hpack-qpack.md`). Transport
concerns — packet framing, congestion control, 0-RTT, connection
migration — belong to `lb-quic`. Alt-Svc advertisement and GOAWAY with
H3_NO_ERROR for graceful shutdown live at the `lb-l7` pipeline.

## MUST clauses we implement

- **Varint framing of type and length (§7.1)** — Every HTTP/3 frame is
  prefixed by two QUIC varints. `decode_frame` calls `decode_varint`
  twice (`frame.rs:85`, `frame.rs:87`). `decode_varint` honours the
  2-bit length prefix and emits 1, 2, 4, or 8-byte integers up to
  `MAX_VARINT = (1 << 62) - 1`, which is what §7.1 inherits from RFC
  9000 §16.
- **Unknown frame types MUST be ignored (§7.2.8)** — Any frame whose
  type does not match the fixed set falls through to `H3Frame::Unknown`
  (`frame.rs:144`). The comment on that match arm quotes the rule.
- **SETTINGS frame layout (§7.2.4)** — Identifier–value pairs encoded
  as concatenated varints; the decoder walks the payload in a loop
  that peels one identifier and one value per iteration
  (`frame.rs:114-125`).
- **CANCEL_PUSH / GOAWAY / MAX_PUSH_ID carry a single varint (§7.2.3,
  §7.2.6, §7.2.7)** — All three are decoded with a single
  `decode_varint` call on the payload (`frame.rs:110-141`).
- **DATA and HEADERS are opaque payloads to the framing layer
  (§7.2.1, §7.2.2)** — We carry the bytes as `Bytes::copy_from_slice`
  references and hand off to the QPACK decoder for headers.
- **Payload-size cap** — The MUST in §10.5 says endpoints SHOULD reject
  frames larger than a reasonable implementation limit; we return
  `H3Error::FrameTooLarge` when the varint payload length exceeds
  `max_payload_size` (default 1 MiB) at `frame.rs:89`, before any
  allocation.
- **Reserved frame types MUST NOT be treated as errors (§7.2.8)** —
  HTTP/3 reserves the HTTP/2 frame types (0x02 PRIORITY, 0x06 PING,
  0x08 WINDOW_UPDATE, 0x09 CONTINUATION) to prevent cross-protocol
  confusion. Our `Unknown` catch-all covers them without raising an
  error, as required.
- **QPACK header compression (§4.1.1)** — Mandatory for all HEADERS
  frames. `QpackEncoder`/`QpackDecoder` in `crates/lb-h3/src/qpack.rs`
  cover RFC 9204 (see `hpack-qpack.md` for the full mapping).
- **CONNECT method semantics carry over from HTTP/2 (§4.4)** — The
  codec accepts `:method: CONNECT`, `:authority`, and no `:scheme` /
  `:path`. Translation logic lives in the bridging tests
  (`tests/bridging_h3_h1.rs`, `tests/bridging_h3_h2.rs`).
- **GOAWAY stream-ID semantics (§5.2)** — GOAWAY in H3 carries a single
  stream ID (the last the sender will process), which is a change from
  H2 where GOAWAY carries last-stream-id plus error code. Our decoder
  reflects this (`H3Frame::GoAway { stream_id }`, `frame.rs:48`).
- **MAX_PUSH_ID for server push gating (§7.2.7)** — Decoded but push
  is disabled at the proxy; receiving a MAX_PUSH_ID never forces us to
  open new push streams.

## Edge cases & security

- **QPACK bomb** — Analogous to HPACK bomb in H2.
  `QpackBombDetector::check:35` enforces both absolute decoded size and
  decoded/encoded ratio. Tests: `tests/security_qpack_bomb.rs`.
- **Oversized frame DoS** — `decode_frame` checks the varint length
  against `max_payload_size` *before* allocating the payload buffer
  (`frame.rs:89`), so a 4 GiB-declared HEADERS frame with no data
  fails fast without OOM.
- **Unknown frame DoS** — The `Unknown` variant keeps the payload so it
  can be logged or metered, but the caller is free to drop it. The
  codec itself will not reject unknown types — required by §7.2.8.
- **0-RTT replay** — Governed at the QUIC layer (see `rfc9000.md`), but
  HTTP/3 methods that are not safe (POST, PATCH, DELETE) MUST NOT be
  accepted over 0-RTT. Enforcement is in the pipeline;
  `tests/security_zero_rtt_replay.rs` exercises the replay-protection
  guard (`lb-security::ZeroRttReplayGuard`).
- **Reserved HTTP/2 frame types on H3 stream** — The codec treats them
  as `Unknown` but a strict server MAY emit `H3_FRAME_UNEXPECTED` per
  §7.2.8; we opt for the permissive "ignore" posture which is also
  compliant.
- **Alt-Svc advertisement (RFC 7838)** — The pipeline injects
  `Alt-Svc: h3=":port"; ma=<seconds>` into HTTP/1.1 and HTTP/2
  responses to trigger HTTP/3 discovery (PROMPT.md §12 item 3).

## Known deviations / TODO

- QPACK's **dynamic table** is not implemented — we use static-table-
  plus-literal representation only. This is fully compliant (dynamic
  tables are optional; §4.1.1 explicitly allows disabling them by
  setting the encoder's `SETTINGS_QPACK_MAX_TABLE_CAPACITY` to zero in
  SETTINGS).
- HTTP/3 prioritization extensions (RFC 9218) are not implemented.
- The framing layer does not currently expose the unidirectional
  control/push/QPACK encoder/decoder streams as separate types — the
  proxy terminates those at the `lb-quic` layer and multiplexes into
  `H3Frame` only for the request/response streams.
- Server push is parsed but never originated.
- Extensible priorities `priority` pseudo-header is not interpreted.
