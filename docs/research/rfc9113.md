# RFC 9113 — HTTP/2

## Summary

RFC 9113 defines the HTTP/2 binary framing layer: a 9-octet frame header,
ten defined frame types, a stream state machine, and the SETTINGS-based
connection negotiation that replaces the ASCII-oriented framing of
HTTP/1.1. In ExpressGateway the RFC maps to the `lb-h2` crate, whose
`decode_frame`/`encode_frame` round-trip every defined type plus the
"unknown frame MUST be ignored" catch-all. HPACK integration,
CONTINUATION flooding, Rapid Reset (CVE-2023-44487), and the H2-to-H1
downgrade smuggling rule from §8.2.2 are all owned by this crate or by
its sibling `lb-security::SmuggleDetector`. HPACK itself is covered in
the companion document `hpack-qpack.md`. The preface, prioritisation
tree, push-promise machinery, and h2c upgrade are addressed at the
pipeline level; the codec stays pure so fuzzers and bounded parsers can
exercise it without a socket.

## MUST clauses we implement

- **9-octet fixed frame header (§4.1)** — `parse_frame_header` reads
  24-bit length, 8-bit type, 8-bit flags, and 31-bit stream identifier
  with the reserved top bit masked (`raw_sid & 0x7FFF_FFFF`,
  `frame.rs:167`), satisfying "the R bit MUST be ignored when received".
- **SETTINGS_MAX_FRAME_SIZE cap (§4.2)** — `decode_frame` rejects any
  frame whose length exceeds the passed `max_frame_size` with
  `H2Error::FrameTooLarge` (`frame.rs:368`). The default constant is
  `DEFAULT_MAX_FRAME_SIZE = 16_384` at `frame.rs:350`, matching §4.2's
  mandatory minimum.
- **Unknown frame types MUST be ignored (§4.1)** — `decode_frame_high`
  returns `H2Frame::Unknown { frame_type, stream_id, flags, payload }`
  instead of erroring (see the comment at `frame.rs:339`). Test
  `unknown_frame_type_ignored` at `frame.rs:655` locks this behaviour.
- **SETTINGS payload structure (§6.5.1)** — Payload length must be a
  multiple of 6 bytes; SETTINGS ACK must carry an empty payload. Both
  are checked at `frame.rs:256,261`, returning `H2Error::InvalidFrame`
  with a descriptive message, as §6.5 requires the recipient to emit
  `FRAME_SIZE_ERROR`.
- **PING payload length (§6.7)** — Exactly 8 bytes; validated at
  `frame.rs:290`.
- **WINDOW_UPDATE length (§6.9)** — Exactly 4 bytes; validated at
  `frame.rs:322`. The top bit (reserved) is masked from the increment
  (`raw & 0x7FFF_FFFF`, `frame.rs:328`), matching §6.9's "R bit must be
  cleared on receipt".
- **PRIORITY length (§6.3)** — Exactly 5 bytes (4-byte dependency +
  1-byte weight); validated at `frame.rs:211`. The exclusive bit is
  decoded separately (`frame.rs:217`).
- **RST_STREAM length (§6.4)** — Exactly 4 bytes; validated at
  `frame.rs:228`.
- **GOAWAY minimum length (§6.8)** — At least 8 bytes (4 last-stream-id,
  4 error-code); validated at `frame.rs:306`. Debug data is captured as
  an opaque `Bytes` slice rather than reinterpreted.
- **HEADERS with PRIORITY flag (§6.2)** — When flag 0x20 is set the
  payload starts with a 5-byte priority prefix which MUST be stripped
  before the header-block fragment is extracted; handled at
  `frame.rs:191`. Test `hpack_encode_decode_roundtrip` (sibling file)
  and `conformance_h2` exercise the path.
- **CONTINUATION without END_HEADERS creates a vulnerability (§4.3)** —
  `ContinuationFloodDetector` caps the number of CONTINUATION frames
  allowed between a HEADERS/PUSH_PROMISE and its terminating
  END_HEADERS, as mandated by the CONTINUATION flood advisory
  (CVE-2024-24549). Regression: `tests/security_continuation_flood.rs`.
- **ENHANCE_YOUR_CALM on control-frame flood (§10.5)** — PROMPT.md
  Module 7 specifies a SETTINGS+PING rate cap (100/10 s). The codec
  surfaces the frame types; the pipeline emits GOAWAY with error code
  0x0b when the limit is hit.
- **H2→H1 header sanitisation (§8.2.2)** — `SmuggleDetector::
  check_h2_downgrade` rejects hop-by-hop headers prohibited in HTTP/2
  (`connection`, `transfer-encoding`, `keep-alive`, `upgrade`,
  `proxy-connection`), rejects any `:pseudo-header` that leaks into the
  H1 projection, and permits `TE` only with value `trailers` (case-
  insensitive). Regression: `tests/security_smuggling_h2_downgrade.rs`.

## Edge cases & security

- **Rapid Reset (CVE-2023-44487)** — `RapidResetDetector` uses a two-
  bucket sliding window (integer-only, scaled by 1000 to avoid FP)
  identical in shape to the NGINX rate-limiter. The boundary attack
  where an attacker floods at window edges is locked down by
  `rapid_reset_boundary_attack_detected` at
  `crates/lb-h2/src/security.rs:253`.
- **HPACK bomb** — `HpackBombDetector::check` caps both absolute
  decoded size and the decoded/encoded ratio; regressions at
  `tests/security_hpack_bomb.rs`.
- **CONTINUATION flood** — The detector resets when `END_HEADERS`
  arrives (`security.rs:138`) so long header blocks that are legitimately
  spread across many frames remain legal.
- **Frame-too-large attack** — Any attempt to announce a length exceeding
  `SETTINGS_MAX_FRAME_SIZE` is rejected before payload allocation
  (`decode_frame:368`), so the decoder cannot be coerced into
  allocating hundreds of megabytes.
- **Padding** — The current codec does not expose PADDED payloads
  (DATA or HEADERS) as a separate path; padding bytes are included in
  the payload slice verbatim and the PADDED flag is folded into the
  "unknown flag, MUST ignore" behaviour. This is a conservative
  deviation (see TODO below).
- **h2c cleartext upgrade (§3.4)** — The 24-byte preface
  `PRI * HTTP/2.0\r\n\r\nSM\r\n\r\n` detection lives in the pipeline,
  not in the codec.

## Known deviations / TODO

- PADDED frames are accepted with the padding bytes inlined in the
  payload; a caller that needs to skip padding MUST do so manually by
  reading the first byte of the payload. A first-class `Padded` variant
  is deferred.
- PUSH is decoded but the server half of PUSH_PROMISE handling is not
  wired up in the proxy pipeline because server push is widely disabled
  in modern clients (§8.4 allows this).
- PRIORITY signalling is parsed but ignored for scheduling — the proxy
  uses a round-robin scheduler per stream, not a weighted dependency
  tree. RFC 9113 §5.3 declares prioritisation signals "advisory", so
  this is compliant.
- `EXTENDED CONNECT` (RFC 8441) is parsed as an ordinary HEADERS frame;
  the `:protocol` pseudo-header is conveyed through the generic header
  list, not a typed field.
- h2c Upgrade is not implemented — PROMPT.md §11 documents the
  cleartext preface but the graceful HTTP/1.1-to-HTTP/2 upgrade path is
  a TLS-ALPN-first deployment model.
