## C. R8 mechanism re-check (the binding four)

1. **Body path bounded-incremental:** YES. Request: peek-then-one-event pump (J2), exactly the verified `write_h1_request`/`H3ReqStreamBody` shape. Response: per-frame re-encode (J1), no `decoded_body`, like `stream_h1_response`/`stream_h2_response`.
2. **Memory bounded by a FIXED in-flight window (not body size, not total-body cap):** YES. Req ≈ `H3_BODY_CHANNEL_DEPTH×(H3_BODY_CHUNK_MAX+MAX_FRAME_HEADER_BYTES)` + 1 in-hand chunk. Resp ≈ `H3_RESP_CHANNEL_DEPTH×(H3_RESP_CHUNK_MAX+H3_FRAME_HDR_MAX)` + 1 in-hand frame. `MAX_REQUEST/RESPONSE_BODY_BYTES` (64 MiB) stay ONLY as DoS abort thresholds passed as `cap`, NOT the memory bound (identical role to H3→H1/H3→H2). Consts tree-confirmed (Section A).
3. **End-to-end backpressure (both directions):** YES, native quiche (no hyper). Response dir: stalled client → `drain_resp_channels` §1.4.3 gate stops pulling → `resp_tx`(8) fills → M-C `send().await` parks → M-C stops `stream_recv` on upstream → quiche withholds `MAX_STREAM_DATA` → upstream send window closes. Request dir: backend stops reading → quiche shrinks M-C send window → `stream_capacity` small → M-C stops pulling `brx` → `brx`(8) fills → M-A stops `stream_recv` on client → client upload paused. Both chains rest on tree-confirmed APIs (`stream_capacity` `lib.rs:6261`, the unchanged §1.4.3 gate `conn_actor.rs:365-440`).
4. **QUIC flow control respected:** YES. Send half consults `stream_capacity` before each request DATA frame; recv half's backpressure is *expressed through* not consuming the upstream stream (so quiche naturally withholds flow-control credit) — no manual window math, only documented quiche 0.28 APIs.

No path reintroduces full-body buffering: `decoded_body` is deleted, no `.collect()`, exactly one bounded pipe per direction. The §6 self-check table holds against the live tree.
