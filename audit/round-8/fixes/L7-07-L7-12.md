Approach (≤500 words):

The two findings collapse into one design by HAProxy's own argument:
operators cannot tune six independent thresholds; HAProxy 3.0 shipped
`tune.h2.fe.glitches-threshold` specifically so the per-connection
"abuse score" is one number. The frame-recv timeout (L7-07) is one
*input* to that score, alongside the existing five detectors
(`RapidResetDetector`, `ContinuationFloodDetector`, `HpackBombDetector`,
`SettingsFloodDetector`, `PingFloodDetector`, `ZeroWindowStallDetector`).

**Design (one structure, two findings closed):**

1. Introduce `GlitchesCounter` in `lb-h2/src/security.rs`:
   ```rust
   pub struct GlitchesCounter {
       window: Duration,
       threshold: u32,                      // default 200
       events: VecDeque<(Instant, u32)>,    // (when, weight)
       per_kind: [AtomicU64; GLITCH_KINDS], // Prometheus surface
   }
   impl GlitchesCounter {
       pub fn record(&mut self, kind: GlitchKind, weight: u32, now: Instant) -> GlitchOutcome;
   }
   enum GlitchOutcome { Allow, Drain }   // Drain → GOAWAY + close
   ```
   `GlitchKind` enumerates: RapidReset (5), Continuation (1),
   Settings (2), Ping (2), HpackRatio (10), ZeroWindowStall (5),
   **FrameRecvTimeout (8)**. Weights derived from HAProxy's
   default cost table (cite the HAProxy commit in the docstring).

2. Each existing detector retains its own threshold for unit-test
   granularity, but calls `GlitchesCounter::record(...)` in
   addition. The connection drains when *either*:
   - an individual detector trips (legacy behaviour, retained), OR
   - the glitches sum-in-window exceeds the consolidated threshold.

3. **L7-07 specific addition — frame-recv timer.**
   In `h2_proxy.rs`, when serving a connection, spawn a per-connection
   `tokio::time::Interval` (interval = `recv_frame_timeout_ms / 4`,
   default 7.5 s for a 30 s deadline). Each polled hyper read updates
   `last_frame_arrival_at: AtomicU64`. The interval task checks
   `now - last_frame_arrival > recv_frame_timeout` → `GlitchesCounter::
   record(GlitchKind::FrameRecvTimeout, 8, now)`. If `Drain` returned,
   abort the connection via hyper's `graceful_shutdown` (already
   wired by REL-2-02).

4. Config:
   - `[runtime].h2_recv_frame_timeout_ms = 30_000` (nginx parity).
   - `[runtime].h2_glitches_threshold_per_window = 200`
   - `[runtime].h2_glitches_window_ms = 60_000`
   The three knobs are not independent of the existing six
   detector knobs — they are the *outer* gate; individual
   knobs become operator-irrelevant in the common case and can be
   left at compile-time defaults.

5. Prometheus surface:
   - `lb_h2_glitches_score{listener}` (current window sum, gauge)
   - `lb_h2_glitches_events_total{listener, kind}` (counter)
   - `lb_h2_glitches_drains_total{listener}` (counter)
   Replaces the six per-detector counters as the primary operator
   chart; existing counters retained for debugging.

Reference pattern: HAProxy `mux_h2.c`'s `h2c->glitches` field +
`tune.h2.fe.glitches-threshold` (default 200; see HAProxy 3.0
release notes and `src/mux_h2.c` `h2_glitches_*` functions).
nginx `http2_recv_timeout = 30s` for the frame-arrival deadline.

**Boundary disclosure:** the connection-drain hook on `Drain`
outcome runs through hyper's `graceful_shutdown` already wired in
`h2_proxy.rs`. No `lb/src/main.rs` changes needed.

Proof:
  - `round8_h2_glitches::frame_recv_timeout_drains_connection` — invariant: an H2 client that sends HEADERS one byte every 31 s triggers a GOAWAY within `recv_frame_timeout + jitter` of the gap; verified by reading the GOAWAY frame off the socket.
  - `round8_h2_glitches::glitches_score_aggregates_across_detectors` — invariant: 10 RapidReset events (50 weight) + 50 Continuation events (50 weight) + 1 HpackRatio (10 weight) + 8 Ping (16 weight) = 126 score (under threshold 200, no drain); add 10 more RapidReset (50) → score 176 still under; add 4 FrameRecvTimeout (32) → score 208, drain emitted.
  - `round8_h2_glitches::individual_detector_still_trips_independently` — regression: a pure rapid-reset flood at 100 events/s still trips the `RapidResetDetector` threshold even if the glitches threshold has not yet been reached. Both paths converge on the same drain primitive.
  - `round8_h2_glitches::counter_resets_after_window` — invariant: events older than `window_ms` are pruned from the score.
  - `round8_h2_glitches::prom_counters_emit_one_kind_per_event` — invariant: each `GlitchesCounter::record(kind, ...)` increments exactly one per-kind counter and exactly once.

Risk / blast radius:
  - The aggregator runs *additively* — connections that previously
    survived under individual thresholds may now drain when the sum
    crosses 200. Default weights are conservative (sum of "noisy
    but legitimate" pre-existing thresholds ~ 100); the 200
    threshold is HAProxy's chosen floor.
  - The frame-recv timer is *per connection*, not per-stream. A
    long-streaming response (server-sent events, very long body)
    that emits zero frames for 30 s would not normally happen
    server-side, but hyper's flow-control may legitimately stall a
    server's send if the client window is exhausted. Operators with
    streaming workloads can raise the timer to 5 min (cite in
    docs).
  - Config knob count grows by three. Tradeoff is acceptable given
    the operator-UX gain of "tune one number".
