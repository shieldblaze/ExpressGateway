# Fix plan — OPS-04 + L4-12 (bundle): drain coordinator (listener cancel arm → in-flight drain → XDP detach)

Owner: div-ops (primary, drain coordinator architecture); div-l4 contributes the XDP detach signature (the implementation lives in `crates/lb-l4-xdp` and is defined in the OPS-01+L4-12+L4-04 bundle).
Findings folded in:
- ROUND8-OPS-04 (high, escalated from medium per cross-review.md §A) — TCP/H1/H1s accept loop has no `tokio::select!` arm on the shutdown token; drain still relies on `JoinHandle::abort()` mid-`accept().await`.
- ROUND8-L4-12 (medium) — XDP detach has no defined ordering vs the L4 fast-path; today the process exits without detaching, leaving a stranded pin.

Lead arbitration applied (cross-review.md §B bundle 5): ONE drain coordinator orchestrates the steps. Two independent code paths (accept-loop abort and XDP detach happening "at process exit") is the bug class. The previous audit had a drain *sequence* and it still missed cases — this plan must enumerate the cases.

Coverage-gap themes cited (the audit-failure-modes that let this slip):
- **Theme 1 — "Verified-Fixed" snapshot of script existence, not capability** (OPS-04). CODE-2-03 migrated spawns through `TaskTracker` and was graded Verified-Fixed; the *accept-loop body* still uses `JoinHandle::abort()`. The status note overclaims scope.
- **Theme 4 — Multi-validator audit handoff** (L4-12). EBPF-2-04 audited the attach mode-ladder; nobody walked the symmetric *detach + multi-program coexistence* predicate.

---

## A. Design — single drain coordinator

`crates/lb-core::Shutdown` already owns a `tokio_util::sync::CancellationToken` and a `TaskTracker`. The drain coordinator is a new method `Shutdown::run_drain(spec: DrainSpec) -> DrainOutcome` that runs the numbered sequence below. The existing `Shutdown::drain` becomes a thin wrapper that constructs a default `DrainSpec`.

```rust
// crates/lb-core/src/shutdown.rs (new)
pub struct DrainSpec {
    pub listener_cancel_deadline: Duration,   // budget for accept-loop cancellation
    pub inflight_drain_deadline: Duration,    // budget for per-conn task completion
    pub xdp_detach_deadline:    Duration,      // budget for XDP detach
    pub readiness_settle:       Duration,      // pre-cancel wait (the lameduck window)
    pub jitter_max:             Duration,      // OPS-02 jitter upper bound
}
pub struct DrainOutcome {
    pub phase_durations: [(DrainPhase, Duration); 6],
    pub listener_outcome: ListenerOutcome,
    pub xdp_detach_outcome: XdpDetachOutcome,
    pub in_flight_remaining: usize,
}
```

The coordinator emits the `shutdown_drain_seconds` histogram (OPS-03), one observation per phase plus a total.

## B. Numbered ordering — every case enumerated

The previous "drain sequence" (`crates/lb/src/main.rs:1925-1984`) was a linear sequence with no enumerated case analysis. This plan enumerates **every** state transition we expect.

### B.1 — Happy path (steady state → SIGTERM → clean exit)

| # | Phase | Action | Bounded by |
|---|---|---|---|
| 1 | `Pre` | Capture `Instant::now()` for the total clock. Log `drain_started`. | — |
| 2 | `MarkDraining` | `probes.set_draining(true)`. Log `set_draining`. | — |
| 3 | `ReadinessSettle` | Sleep `spec.readiness_settle` (OPS-11 will raise this default; see OPS-11 plan). Probes return 503 from now on. | `spec.readiness_settle` (default raised from 1000 to 2 × kubelet probe period; OPS-11). |
| 4 | `ListenerCancel` | Trigger listener cancel: cancel the `listener_cancel_token` (a *new* child of `shutdown_token`, owned by the coordinator). Every per-listener accept loop selects on this token (cooperative — OPS-04 fix). Await each listener task with `tokio::time::timeout(spec.listener_cancel_deadline, JoinAll(handles))`. On timeout: `JoinHandle::abort()` per listener and record `listener_outcome = TimedOut`. | `spec.listener_cancel_deadline` (default 500 ms — listener accept loops are cooperative once the OPS-04 fix lands). |
| 5 | `InFlightDrain` | `tracker.close()` then `tracker.wait()` with a `tokio::time::timeout(spec.inflight_drain_deadline)`. Per-conn tasks observe `shutdown_token.cancelled()` in their existing `biased select!` (already cooperative). Apply OPS-02 jitter: each per-conn cancel arm sleeps `Duration::from_millis(rand::thread_rng().gen_range(0..spec.jitter_max.as_millis()))` before propagating into the protocol layer. | `spec.inflight_drain_deadline` (default `runtime.drain_timeout_ms`, default 10 s; per-listener override per OPS-10). |
| 6 | `XdpDetach` | If XDP is enabled: call `XdpLoader::detach()` (see OPS-01+L4-12+L4-04 plan Part B for the signature). Bounded wait with `tokio::time::timeout(spec.xdp_detach_deadline)`. On timeout: log `xdp_detach_timeout`, increment `xdp_detach_total{result="dirty_post_query"}`, and proceed (we have already exited the listen path; the XDP program will linger but the new process can recover it via the stale-self path in OPS-01+L4-12+L4-04 §B.2 step 1c). | `spec.xdp_detach_deadline` (default 1 s). |
| 7 | `Done` | Log `drain_completed`, observe histograms, return `DrainOutcome`. | — |

### B.2 — Case enumeration (the cases the previous audit missed)

For each case, label whether OPS-04+L4-12 today handles it correctly (★ = currently broken, ✓ = currently OK, ◆ = currently dirty but bounded).

| Case | Today | After this fix |
|---|---|---|
| C-1: SIGTERM arrives with no active connections, no in-flight accept | ✓ | ✓ |
| C-2: SIGTERM arrives during a pending `accept().await` | ★ JoinHandle::abort cancels at next poll (OK), but next case below | ✓ Phase 4 cancels cooperatively, no unmatched accepted-fd is possible because the accept future has no in-progress post-accept work. |
| C-3: SIGTERM arrives in the **gap** between `accept()` returning Ok and the per-conn task being spawned via `tracker.spawn()` (admission gate, semaphore acquisition, IP cap increment) | ★ `JoinHandle::abort` interrupts mid-step; accepted socket leaks; per-IP counter increments without decrement; spawn never reached. | ✓ Phase 4 uses cooperative cancel — the loop body checks `listener_cancel_token.is_cancelled()` immediately after `accept()` returns Ok. If cancelled, the accepted fd is closed via Drop, no spawn, no counter increment. The select! arm covers the *future*, the post-accept check covers the *synchronous tail*. **Both required.** |
| C-4: SIGTERM during slow handshake (TLS) — per-conn task is spawned but `ServerConfig::accept` is mid-handshake | ◆ task observes `shutdown_token.cancelled()` in its `biased select!` (already cooperative) but Rustls handshake future may not check. | ✓ Phase 5 timeout + per-conn drop. Rustls handshake future is dropped on cancel; socket closes via Drop. |
| C-5: SIGTERM with long-lived H2/H3 stream (gRPC bidi, SSE) | ◆ stream is severed at `drain_timeout_ms`. | ◆ same — but now `drain_timeout_ms` is per-listener (OPS-10) and the operator can set the streaming listener to 5min. Documented. |
| C-6: SIGTERM with WebSocket | ◆ stream is severed at `drain_timeout_ms`. | ◆ same as C-5. Documented. |
| C-7: SIGTERM and `tracker.wait()` times out (some task is stuck) | ★ proceeds to process exit; XDP program lingers; no metric. | ✓ phase 5 records `in_flight_remaining` count; phase 6 still runs XDP detach. |
| C-8: SIGTERM and XDP detach itself hangs (unlikely, but possible if kernel bug) | ★ process never exits → systemd `TimeoutStopSec` kills with SIGKILL → XDP lingers stranded. | ✓ phase 6 has its own timeout; we log + emit metric + proceed to process exit. The XDP-stale-self code in OPS-01+L4-12+L4-04 §B.2 recovers on next startup. |
| C-9: SIGTERM during XDP attach itself (startup race) | ★ undefined. | ✓ XDP attach completes (or fails) before `Server::run_forever` even enters the steady state; coordinator's `MarkDraining` cannot run until startup completes. We add an explicit `Shutdown::request_drain_after_startup()` so a SIGTERM during startup is queued. |
| C-10: Two SIGTERMs in quick succession | ★ second SIGTERM calls drain twice; race on `set_draining`. | ✓ `run_drain` is idempotent via an `AtomicBool`. Subsequent calls log + return the first outcome. |
| C-11: SIGTERM while `set_draining` already true (admin set draining via `/admin/drain`) | ★ same race as C-10. | ✓ idempotent. |
| C-12: Process crashes mid-drain (panic in a per-conn task) | ★ partial state; pins stranded. | ◆ panic-handler in `tracker.spawn` already catches per-task panics. Drain coordinator panic is the new failure mode: we install a `scopeguard` that runs XDP detach in `drop()` even on panic. |
| C-13: Drain initiated by `/admin/drain` (no signal); operator wants to keep the process running | ★ today this is "set_draining + wait_for_inflight + ???"; XDP detach would be wrong here (operator wants to *un-drain*). | ✓ `DrainSpec::xdp_detach_deadline = None` for admin-drain; coordinator skips phase 6. An `/admin/undrain` endpoint unsets the bit (out of scope for this plan; tracked separately). |
| C-14: Drain initiated and one listener accept loop deadlocked (cooperative cancel ignored — e.g. a future panic) | ★ JoinHandle::abort eventually clears it. | ✓ phase 4 has `JoinHandle::abort` as the fallback after the cooperative timeout. |
| C-15: SIGTERM with **zero** listeners bound (degenerate config) | ✓ | ✓ |

C-3 is the canonical case OPS-04 names: the per-IP counter drift across drains. Today this happens *every drain* because the cancel arm is missing. The previous audit had a "drain sequence" but no per-case table — this is the artefact the previous audit needed and lacked.

## C. Code-level changes

1. **`crates/lb-core/src/shutdown.rs`**: add `DrainSpec`, `DrainOutcome`, `DrainPhase`, `ListenerOutcome`, `XdpDetachOutcome`, `Shutdown::run_drain`. The existing `Shutdown::drain(timeout)` becomes `Shutdown::run_drain(DrainSpec::legacy(timeout))`.
2. **`crates/lb/src/main.rs:2179-2211`**: rewrite the accept loop body:
   ```rust
   loop {
       let (client_stream, client_addr) = tokio::select! {
           biased;
           () = state.listener_cancel_token.cancelled() => {
               tracing::info!(address = %bind_addr, "listener cancelled by drain");
               return Ok(());
           }
           res = listener.accept() => match res {
               Ok(ok) => ok,
               Err(e) => { /* existing accept-error backoff (REL-2-10) */ continue; }
           }
       };
       // SYNCHRONOUS-TAIL CHECK for case C-3:
       if state.listener_cancel_token.is_cancelled() {
           tracing::debug!("accepted socket dropped post-cancel");
           drop(client_stream);
           continue; // or break — same outcome since select fires next iter
       }
       // ... existing admission gate, semaphore, spawn ...
   }
   ```
3. **`crates/lb/src/main.rs:1925-1984`** (drain block): replace with a call to `Shutdown::run_drain(DrainSpec::from_runtime_config(&cfg.runtime, &cfg.listeners))`. Move the metric emission inline.
4. **`crates/lb/src/main.rs:1942-1944`** (`for h in &listener_handles { h.abort(); }`): delete — replaced by phase 4 of `run_drain`.
5. **`crates/lb-l4-xdp/src/loader.rs`**: add `XdpLoader::detach()` per OPS-01+L4-12+L4-04 §B.2 step 4.
6. **Shutdown token hierarchy**:
   - `shutdown_token` (root, the existing one).
   - `listener_cancel_token = shutdown_token.child_token()` — fired by phase 4. Listeners select on this.
   - Per-conn tasks continue to select on `shutdown_token` directly (so the conn cancel fires during phase 5, not phase 4).
7. **Metrics emitted**:
   - `shutdown_drain_seconds{phase, outcome}` histogram (also fulfills OPS-03 — see OPS-03 plan; this plan does not duplicate the histogram setup).
   - `shutdown_listener_cancel_timeout_total` counter.
   - `shutdown_inflight_drain_timeout_total` counter.
   - `xdp_detach_total{result}` counter (per OPS-01+L4-12+L4-04 plan).

## E. Verification — what "Verified-Fixed" looks like

- OPS-04: a verifier (div-l7 or proto) walks `crates/lb/src/main.rs` accept loop and confirms (a) the select on `listener_cancel_token` exists with `biased`, (b) the post-accept synchronous-tail check exists, (c) `JoinHandle::abort` on listeners is gone or only used as a phase-4 timeout fallback.
- L4-12: a verifier (div-l4) walks `XdpLoader::detach` and confirms it (a) reads `bpf_xdp_query` before destroy, (b) re-queries after destroy and bumps `xdp_detach_total{result="dirty_post_query"}` if non-zero, (c) is invoked unconditionally in phase 6 of `run_drain`.
- The `shutdown_drain_seconds` histogram has at least one observation per phase in `/metrics` after a drain (which OPS-03 also requires).

## F. Risk / open questions

- `tokio_util::sync::CancellationToken::child_token` semantics: cancelling the parent also cancels the child. We want the *reverse* (cancel child without cancelling parent) — child_token gives us that. Confirmed.
- Phase 4 deadline (500 ms): if a listener's accept loop has a deeply-nested future that doesn't check cancellation, abort fallback fires. The `biased select!` placement at the *top* of the loop guarantees responsiveness per iteration; a stuck listener is only possible if the accept future itself never returns, which is a kernel pathology not a config issue.
- The per-conn jitter (OPS-02) is included in phase 5 because the OPS-02 fix is in the per-conn `biased select!` cancel arm — semantically part of the in-flight drain.
