## A. The gate, philosophically

A doc-lint gate is a contract between the operator-facing prose and the source. We extend that contract two ways:

1. **Positive assertion** (Theme 3): every fact the docs state about defaults / behaviour / metric names / capability sets must have a source-of-truth that the gate imports at lint time. Drift = compile error / lint failure.
2. **Audit-of-audit** (Theme 1): every `Verified-Fixed(<sha>)` claim in `audit/**/round-*-review.md` must reference a SHA whose diff actually closes the recommendation. The gate parses both the claim and the diff and asserts overlap.

## B. Tier 1: positive-assertion model for operator docs

### B.1 — Rust-side: `cargo-doc-lint` binary

Replace `scripts/ci/doc-lint.sh` (or wrap it) with a Rust binary at `xtask/src/bin/doc_lint.rs` (or a workspace-member `tools/doc-lint`):

// pseudocode
fn main() -> Result<()> {

    assert_runbook_default_matches::<{ DEFAULT_DRAIN_TIMEOUT_MS }>("drain_timeout_ms", "RUNBOOK.md")?;
    assert_runbook_default_matches::<{ DEFAULT_READINESS_SETTLE_MS }>("readiness_settle_ms", "RUNBOOK.md")?;

    assert_capability_set_matches("DEPLOYMENT.md", &["CAP_NET_BIND_SERVICE", "CAP_NET_ADMIN", "CAP_BPF"])?;

## C. Tier 2: audit-of-audit gate (the main lead-arbitrated piece)

### C.1 — What the gate parses

The gate walks `audit/**/round-*-review.md` and for every line matching:
```
Status:   Verified-Fixed(<verifier=NAME, >?author-sha=<SHA>(\+<SHA>)*)
```
(also `Verified-Fixed-Partial` and `Verified-Fixed-Partial(verifier=..., author-sha=...)`)

extracts:
- The finding ID (the `### ID — title` line above).
- The `Recommendation:` block content.
- The author SHA(s).
- The HTML comment `<!-- ... -->` immediately after the status (which contains the verifier's justification).

### C.2 — What the gate verifies

For each parsed claim, the gate:

1. **SHA exists.** `git cat-file -e <sha>^{commit}`. If not, error.
2. **SHA touched the relevant path.** Each finding has a `Location:` field. The gate runs `git show --stat <sha>` and asserts that at least one path in the diff matches at least one path in `Location:` (substring match on the file portion; not line numbers). If not, error.
3. **Content match against recommendation.** This is the new piece. The recommendation block typically contains either:
   - **Metric names** (e.g. `shutdown_drain_seconds`, `accept_inflight`, `xdp_attach_query_total`): the gate greps the SHA's diff for the literal name. If not present in the diff *and* not already present in the parent, error.
   - **Function / struct names** (e.g. `LabelBudget::check`, `Shutdown::run_drain`, `classify_accept_error`): same — grep the diff. If the name is mentioned in the recommendation, it must appear in the diff at an `+` line (added).
   - **Files** (e.g. `packaging/expressgateway.service`, `audit/ebpf/verifier-logs/*.log.committed`): the gate runs `git ls-tree -r <sha> -- <path>` and asserts the file exists at that SHA. **This is the L4-10 case** — `git ls-tree -r ffde98c -- audit/ebpf/verifier-logs/` would have returned only `README.md`, exposing the no-op.
4. **Verifier comment matches.** The HTML comment field `verifier=<NAME>` must be one of the validator names (sec, code, ebpf, rel, proto, div-ops, div-l4, div-l7, verify). The gate enforces a whitelist.

### C.3 — The L4-10 case, explicitly

For `EBPF-2-07` (the L4-10 finding's subject):
- Recommendation block contains: `audit/ebpf/verifier-logs/{5.15,6.1,6.6}.log.committed`.
- Status: `Verified-Fixed(ffde98c)`.
- Gate runs `git ls-tree -r ffde98c -- audit/ebpf/verifier-logs/` → only `README.md`.
- Gate errors: `EBPF-2-07 Verified-Fixed(ffde98c) but audit/ebpf/verifier-logs/{5.15,6.1,6.6}.log.committed missing in the SHA's tree`.

This is the failure mode the lead asked for ("Verified-Fixed snapshots existence, not capability").

### C.4 — Resolution of REL-2-09 and similar partials

The gate must also handle `Verified-Fixed-Partial`. For these, the comment block typically contains a "deferred" or "remaining" sentence. The gate:
- Asserts the comment matches the regex `(deferred|remaining|partial|not.*yet|todo)` — if not, the partial tag is suspect and the gate warns.
- Parses the partial-disclosure sentence and emits it to a new `audit/round-*-partial-disclosures.md` digest. The digest is the artefact a future round must re-walk.

### C.5 — The recommendation parser

The recommendation block is free-form Markdown. The parser uses heuristics:
- Code spans (`` `name` ``) are *candidate* assertions (metric / function / file names).
- Italic / bold spans are *not* assertions (typically prose).
- File paths matching `crates/**/*.rs`, `scripts/**`, `packaging/**`, `audit/**` are file assertions.
- Identifiers matching `[a-z_]+(::|\.)[a-zA-Z_]+` are function/method assertions.
- Identifiers matching `[a-z_]+_total|[a-z_]+_seconds|[a-z_]+_count` are metric assertions.

If a recommendation's candidate set is empty (the parser found nothing assertable), the gate warns and asks the human to add an explicit machine-readable `Acceptance:` field below `Recommendation:`. Going forward (Round 9+), all findings should carry an `Acceptance:` block.

Add to `audit/round-8/findings/_template.md` (and propagate to future rounds):
```
### <ID> — <title>
Reference: ...
Our equivalent: ...

Severity: ...
Status:   ...

Divergence: ...
Impact:    ...
Recommendation:
1. ...

Acceptance:
- file: <path-or-glob> must appear in the closure SHA's tree
- function: <ident> must appear at a + line in the closure SHA's diff
- metric: <name> must appear in /metrics at the closure SHA
- test: <test-name> must pass at the closure SHA
```

The `Acceptance:` block is machine-readable; the gate parses it directly without needing the recommendation-block heuristics.

## I. Risk / open questions

- The recommendation-block parser is heuristic; some legitimate recommendations may not parse cleanly. Mitigation: the `Acceptance:` field is the canonical form going forward; the parser is the bridge for historical findings.
- The audit-of-audit gate adds runtime cost (one `git show` per finding × ~80 findings = ~5s on a warm cache). Acceptable for CI.
- `git ls-tree -r <sha> -- <glob>` does not support `**`; we use `--` plus an explicit list of paths and rely on substring match.
- Verifier whitelist may need updates as new validators are added; the gate emits a warning (not error) on unknown verifier names so adding a new validator doesn't immediately break CI.
