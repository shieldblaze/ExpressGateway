## Part 1 — OPS-08: SBOM regen via `cargo-cyclonedx`

### 1.1 — CI integration

`.github/workflows/ci.yml` add or extend the `audit` / sibling job:
```yaml
sbom:
  runs-on: ubuntu-latest
  steps:
    - uses: actions/checkout@v4
    - uses: dtolnay/rust-toolchain@stable
    - name: Install cargo-cyclonedx
      run: cargo install cargo-cyclonedx --version "^0.5" --locked
    - name: Generate SBOM
      run: cargo cyclonedx --format json --output-pattern bom --override-filename audit/sbom.json
    - name: Diff against committed
      run: git diff --exit-code -- audit/sbom.json
    - name: Validate schema
      run: |
        cargo install cyclonedx-cli --locked
        cyclonedx-cli validate --input-file audit/sbom.json --schema-version 1.5
```

The diff-against-committed makes regenerate-and-commit the contract: every PR that changes `Cargo.lock` must also commit the updated SBOM, or CI fails. This is the OPS-08 "audit didn't defer to CI" stance applied.

### 1.2 — Fallback

If `cargo-cyclonedx` is not installable in the CI runner image (rare; the crate is well-maintained), fall back to a container action `cyclonedx/gh-cargo-cyclonedx@v1` (verify pin at plan-approval). The current `manual-fallback` tool stays in-tree as `tools/sbom-fallback/` but is no longer invoked from CI — kept only for local generation on platforms where the official tool fails.

### 1.3 — Regenerate the committed SBOM

One-shot: run `cargo cyclonedx --format json --output-pattern bom --override-filename audit/sbom.json` locally; commit the result. The `metadata.tools` field will now read `[{vendor: "CycloneDX", name: "cargo-cyclonedx", version: "0.5.x"}]` and `metadata.timestamp` will be `Utc::now()` at generation time.

### 1.4 — Tests / verification

- CI gate passes on the regenerated SBOM.
- `jq '.metadata.tools[0].vendor' audit/sbom.json` → `"CycloneDX"` (or `cyclonedx-rust-cargo`, depending on the tool's self-reporting).
- `cyclonedx-cli validate` passes.

### 2.2 — `--healthcheck` subcommand (Phase 2, future)

`crates/lb/src/main.rs` CLI arg parser: add `--healthcheck` flag. When set, the binary:
- Reads the admin bind address from config (or defaults to `127.0.0.1:9090`).
- Opens a TCP connection to `<admin>/livez`.
- Exits 0 on HTTP 200, 1 on anything else.
- Does not start any listeners.

Then enable `HEALTHCHECK --interval=30s --timeout=3s --start-period=10s CMD ["/usr/local/bin/expressgateway", "--healthcheck"]`.

We do not implement this in the OPS-12 plan body — it's a small but cross-cutting CLI change. The plan defers to a follow-up issue.

### 2.3 — EXPOSE list

Today: `EXPOSE 80 443 8080 9090`. Drop `80 443 8080` — the binary does not bind those by default. Keep `9090` (admin/metrics) since that IS the binary's documented default. Add a comment:
```dockerfile
# EXPOSE only the binary's default bind set. Operator listeners (80/443/etc)
# are configured via [[listeners]] in the TOML and EXPOSE'd by the operator's
# deployment manifest (K8s podSpec, Compose service ports).
EXPOSE 9090
```
