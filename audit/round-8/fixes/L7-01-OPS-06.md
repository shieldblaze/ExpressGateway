Approach (≤500 words):

**Part A — defer the 101 (Pingora GHSA-xq2h-p299-vjwv / Envoy GHSA-rj35-4m94-77jh shape):**

Today `handle_ws_upgrade` synchronously builds the `101 Switching Protocols`
response and returns it from `ProxyService::call`, then a `tokio::spawn`-ed
`run_h1_ws_upgrade_task` does the upstream dial. The reference proxies
(Pingora, Envoy) reverse that order: dial the upstream and complete the
upstream WS handshake first, only then emit `101` to the client.

Restructure:
1. In `handle_ws_upgrade`, before building any response, perform the upstream
   dial: `pool.acquire_async(backend_addr)` + `client_async_with_config(...)`
   under a bounded `tokio::time::timeout` (re-use the existing pool timeout
   knob).
2. On upstream-handshake **success**: capture the upstream's response headers
   (Sec-WebSocket-Accept, Sec-WebSocket-Protocol, Sec-WebSocket-Extensions)
   and *only then* return a `Response::builder().status(101)` to hyper, with
   those headers reflected from the upstream. Hyper's `on_upgrade` future is
   awaited as today; the spawned task no longer needs to dial — it only needs
   to splice bytes.
3. On upstream-handshake **failure / timeout / non-101**: return a
   synchronous `502 Bad Gateway` (or `504 Gateway Timeout`) — we are still in
   H1 mode because no `101` was emitted; the client connection remains
   keep-alive-eligible.
4. Drop the synchronous `101` write entirely. The `OnUpgrade` future is only
   resolved *after* upstream success.

Reference pattern: Pingora GHSA-xq2h-p299-vjwv fix in `pingora-proxy/src/proxy_h1.rs` — `upgrade_resp` is awaited from the upstream and gated on its status before any response is built for the client. Envoy `Http::WebSocket::WsHandlerImpl` does the same.

**Part B — traceparent propagation through the upgrade path:**

The library at `crates/lb-observability/src/tracing_propagation.rs` already
ships `extract_parent`, `inject_into`, `span_name`, and `TraceContext`. Zero
L7 callsites use it. Wire-in:

1. At the *top* of `ProxyService::handle` (line ~551), call
   `lb_observability::tracing_propagation::extract_parent(req.headers())` and
   open `tracing::info_span!(span_name::REQUEST_H1, trace_id = %ctx.trace_id_hex(), parent_id = %ctx.parent_id_hex(), http.method, http.target, sni, listener)`.
   `sni` is already in scope at that depth (passed in from
   `serve_connection_with_cancel_sni`); `listener` is the `Arc<String>` we
   already plumb. **The sni half is small and asked for in the brief:
   extract sni + traceparent at TLS-accept, propagate into the upgrade
   handler** — the sni is already captured at TLS-accept; we just need to
   thread it into the `info_span!` and into the WS upgrade task so the
   spawned task instruments with the same parent span.
2. Before issuing the upstream dial in `handle_ws_upgrade`, call
   `inject_into(req_to_upstream.headers_mut(), &child_ctx)` so the upstream
   sees a fresh `traceparent` whose parent is our LB span.
3. The spawned `run_h1_ws_upgrade_task` must be `.instrument(span.clone())`
   so its tracing events nest under the request span.
4. On status return, record `http.status_code` on the span via
   `tracing::Span::current().record(...)`.

**Boundary disclosure:** the OTLP *exporter* wire-up
(`[observability].otlp_endpoint`) lives in `lb/src/main.rs` (div-ops's
house). This plan does not require exporter changes — `tracing` events
will be visible via any layer the runtime composes. div-ops owns the
exporter knob.

Proof:
  - `round8_ws_upgrade_defer::client_sees_502_when_backend_rejects_ws` — invariant: when the upstream backend returns `200` to the WS handshake (deliberate non-`101`), the client receives `502` and the wire never carries `101`. Uses a faux backend `TcpListener` that consumes the handshake and writes `HTTP/1.1 200 OK\r\n\r\n`.
  - `round8_ws_upgrade_defer::client_sees_504_on_dial_timeout` — invariant: with `[upstream].connect_timeout_ms = 50` and a non-listening backend, client sees `504 Gateway Timeout`, no `101`.
  - `round8_ws_upgrade_defer::no_smuggled_request_forwarded_on_failure` — invariant: client sends `Upgrade: websocket\r\n\r\nGET /smuggled HTTP/1.1\r\nHost: x\r\n\r\n`; backend rejects WS; assert the `/smuggled` request never reaches a separate sentinel backend (Pingora reproducer shape).
  - `round8_traceparent_propagation::upstream_receives_child_traceparent` — invariant: client sends `traceparent: 00-<32hex>-<16hex>-01`; the inner request to the upstream contains a `traceparent` whose trace-id matches the client's and whose parent-id matches the LB-emitted span id (i.e. *not* the client's parent-id verbatim).
  - `round8_traceparent_propagation::ws_upgrade_inherits_span` — invariant: a successful WS upgrade emits at least one tracing event from inside the spawned splice task and the event's `trace_id` field equals the request's trace_id (verified via a `tracing_subscriber` layer that snapshots events).

Risk / blast radius:
  - **Part A**: the WS upgrade path is on the hot edge for any operator running ExpressGateway in front of a WS workload. The behaviour change is operator-visible: failed upstream WS handshakes now produce `502/504` instead of `101 then silent-close`. This is the *correct* change (mirrors Pingora / Envoy / nginx) but external WS clients that *currently* rely on the bug (i.e. see `101` and only later notice no traffic) will surface failures earlier. Document in CHANGELOG.
  - **Part A** moves the dial+handshake from a `tokio::spawn`-ed task into the request handler's await chain. This increases per-request latency for WS upgrades by exactly the upstream handshake RTT — this is unavoidable and what every reference proxy does.
  - **Part B**: low-risk; the library is well-tested. Worst case is a spans-not-emitted scenario which is silent.
  - **Pool acquisition**: hyper's `on_upgrade` future must remain valid across the new await. Verify with a smoke test that does not regress the success path.
