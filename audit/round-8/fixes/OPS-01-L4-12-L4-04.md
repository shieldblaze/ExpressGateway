## Part B — L4-12: clean XDP detach + bounded operator-perceived outage window

### B.1 — Current behaviour (failure mode A)

`crates/lb-l4-xdp/src/loader.rs:613-630` calls `xdp.attach(ifname, mode.to_flags())`. If a program is already attached (our own pin from a previous crashed process, or a foreign program), aya passes through the kernel's default behaviour and the attach returns `EBUSY`. The new process loops in `attach_with_fallback` through Drv→Skb→Generic and exits.

Operator-perceived outage: **unbounded.** The kernel still has the previous process's XDP program pinned and active; until the operator runs `ip link set dev <iface> xdpgeneric off` (or reboots, or unpins the bpffs object), no replacement instance can attach.

### B.2 — Proper-detach behaviour (this plan implements)

1. **Two-step attach (replaces unconditional attach):**
   ```rust
   // pseudo: in XdpLoader::attach
   let q = bpf_xdp_query(ifindex)?;
   match q {
       None => xdp.attach(ifname, mode.to_flags()),   // virgin interface
       Some(info) if info.prog_id == my_last_prog_id => reuse_existing_link(info),
       Some(info) if info.prog_tag == EXPRESSGATEWAY_PROG_TAG => {
           // It's a stale pin from a previous instance. Detach then reattach.
           detach_by_prog_id(ifindex, info.prog_id)?;
           xdp.attach(ifname, mode.to_flags())
       }
       Some(info) => Err(XdpLoaderError::ForeignProgramAttached(info.prog_id)),
   }
   ```
2. **`attach_replacing(ifname, mode, old_prog_fd)` API** that OR-in `BPF_F_REPLACE` for a single atomic self-replace (used by a future "soft reload" path). Not used in v1 attach but the API ships so the test surface is real.
3. **Clean drop on SIGTERM:** the drain coordinator (see OPS-04+L4-12 bundle for the ordering) calls `XdpLoader::detach()` BEFORE process exit. Detach must:
   a. Read the current `prog_id` via `bpf_xdp_query`.
   b. If `prog_id == ours`, call `bpf_link__destroy` (or aya equivalent) for our link handle.
   c. Re-query: `bpf_xdp_query` must return `None`. If not, log `xdp_detach_post_query_dirty_total` and proceed with process exit anyway.
4. **`xdp_attach_query_total{result}` metric** with labels `ok|foreign|stale_self|none`.
5. **`xdp_detach_total{result}` metric** with labels `ok|already_gone|dirty_post_query`.

### B.3 — Bounded operator-perceived outage window

Without proper detach: outage is unbounded (operator must manually intervene).

With this plan's detach:
- Self-replace path (`attach_replacing` with `BPF_F_REPLACE`): outage window = **0** (atomic at kernel level, the very thing `BPF_F_REPLACE` was designed for).
- Detach-on-SIGTERM + replacement process: outage window = **(time between detach syscall and new attach syscall)** which is bounded by `drain_timeout_ms + readiness_settle_ms + process restart time` (10 + 1 + ~0.5 s ≈ 12 s). The replacement process must finish its `XdpLoader::load_from_bytes_pinned` before attach succeeds.
- Crash path (old process exits without detach): bounded by detach-stale-self code in step 1 above — new process detects the previous prog_id, detaches it itself, then attaches. Outage window = **(crash detection time) + (one detach syscall) + (one attach syscall)** ≈ <1 s assuming systemd auto-restarts.

The bound matters because OPS-01 (deletion of zero-downtime claim) needs a *truthful* upper bound to advertise. The corrected README sentence in A.2 above does not advertise a specific number, but the RUNBOOK Drain section will now be able to add "typical operator-perceived L4 outage during a replace is <12 s in the SIGTERM-replace flow and <1 s in the crash-recovery flow."

## Part C — L4-04: atomic backend-table publication (DEFERRED, this plan documents only)

This plan does **not** implement Part C. L4-04 is Pillar-4b-3-or-later. We document the deferral and the acceptance criterion here so it is not punted to "after first incident."

### C.1 — Acceptance criterion (for the future PR)

The replacement design is per the L4-04 finding's recommendation:
1. Per-VIP `BackendTable` map keyed by VIP, value `{generation: u32, entries: [BackendEntry; MAX_PER_VIP]}`. Atomic publication = a single `map_update_elem` of the whole struct.
2. BPF program looks up `BackendTable[vip]` then indexes `entries[backend_idx % N]`; optionally consults `CONNTRACK` for stickiness.
3. Daisy-chain "previous slot" per Unimog lesson 3 so in-flight flows during a swap reach the previous backend.
4. Integration test: 1k-rebalance under simulated 100k packets/sec, assert no flow sees a partial table.

### C.2 — Documentation in v1

Add `BACKENDS.md` (or a section in `CONFIG.md`) with a "Backend hot-swap (Pillar 4b-3)" subsection. Current behaviour: best-effort, walk-the-conntrack-map. Operator-visible: rebalance windows >100ms may exhibit per-flow inconsistency. Pillar 4b-3 future work: atomic publication.

This is the same pattern as the README claim deletion: the v1 doc tells the truth about today's behaviour, and the future-work entry pre-stages the acceptance test.

## Open questions / risks

- The `EXPRESSGATEWAY_PROG_TAG` constant: we need a stable identifier for "this is our XDP program" that survives recompile. Approach: bake a build-time `prog_tag` const (SHA of `ebpf/src/main.rs` source + lb version), exposed via a `tag` map. Verifier-log gate from OPS-09 will catch tag drift.
- `bpf_xdp_query` is available from kernel 5.9+; we already require ≥5.8 per SEC-2-11. If an operator runs 5.8 exactly, we fall back to "best-effort attach, no query" — log a `warn!` once at startup. (5.8 is the documented floor; not a regression.)
- `BPF_F_REPLACE` available from kernel 5.6+; fine.
