### ROUND8-L4-01 — backend_idx=0 / zero-IP sentinel
Commit: 7f6c9a55. Proof: `round8_backend_sentinel.rs` 7/7 PASS.

eBPF data-plane sentinel guard verified present at
`ebpf/src/main.rs:791-794` (IPv4) and the IPv6 mirror: after
`incr_stat(STAT_CT_HIT_V4)` and BEFORE `rewrite_v4`, `entry.backend_ip == 0
|| entry.backend_port == 0` returns `XDP_PASS` + `STAT_BACKEND_UNPOPULATED`
(slot 10). This catches ANY zero-shaped entry regardless of construction
path, including a partial-population race that wrote bytes but not
`backend_ip`/`backend_port` — adversarial bypass attempt: a half-written
all-zero or zero-ip/zero-port entry yields XDP_PASS, NOT
`XDP_TX 0.0.0.0:0`. The race that the finding describes is closed at the
data plane.

Caveat (plan deviation, non-blocking): plan step 2 said make
`BackendEntry::new` fallible (`Result`). Implementation instead ADDED
`try_new` and kept infallible `new` for back-compat (pod_padding tests
pass zero shapes deliberately). The userspace constructor is therefore a
softer gate than planned, but the eBPF sentinel (the real defence) is
unconditional. Second caveat: planned `sim_xdp_zero_backend_pass_not_tx`
data-plane simulator test was NOT shipped; the eBPF guard is verified by
code inspection only, not an executable test on this branch.

Verdict: **Verified-Fixed** (caveats non-blocking — data-plane guard is
the load-bearing defence and is correct).
Blocking for prod: NO.

Commit: 43d250ee. Proof: `round8_synflood_cap.rs` 4/4 PASS.

ADVERSARIAL CHECK (the special-attention item — does the cap throttle
established flows?): NO. `is_under_flood()` (`main.rs:464-490`) is called
ONLY inside the `None =>` (conntrack-MISS) arm of the CONNTRACK lookup
(`main.rs:777-780`). The CT-HIT arm (`Some(v)` -> `STAT_CT_HIT_V4` ->

`bpf_ktime_get_ns()`, `saturating_add`, window reset on rollover. On cap
exceed it returns XDP_PASS WITHOUT the STAT_PASS "populate CT" signal, so
the LRU is not thrashed — exactly the Katran lesson-4 behaviour.

Verdict: **Verified-Fixed**.
Blocking for prod: NO.
