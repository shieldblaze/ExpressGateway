## ROUND8-L7-01 + OPS-06 ‚Äî Premature 101 (Pingora GHSA-xq2h-p299-vjwv / Envoy GHSA-rj35-4m94-77jh, CVSS 9.3)

- **Fix sha**: `6253ad9a`
- **Plan adherence**: YES. `handle_ws_upgrade` restructured exactly per plan Part A ‚Äî `dial_upstream_ws` is `.await`-ed under `tokio::time::timeout(self.timeouts.header, ..)` BEFORE any client response is built. Single `Response::builder().status(SWITCHING_PROTOCOLS)` at h1_proxy.rs:1299 is unreachable until the timeout future resolves `Ok(Ok(ws))` at line 1254. Failure paths return 502 (refused/unreachable) / 504 (timeout) with the wire still H1. `run_h1_ws_splice_task` is splice-only. Part B (traceparent): `RequestTrace` opens `lb.l7.request` span, child traceparent injected onto the upstream WS handshake builder, splice task `.instrument()`-ed (no Entered guard across await ‚Äî correct).
- **Proof tests**: `round8_ws_upgrade_defer` 4/4 PASS, `round8_traceparent_propagation` 1/1 PASS, `round8_drain_15case` 16/16 PASS.
- **Bypass attempts**:
  - *Race: slow upstream handshake leaks a 101 before the upstream 101* ‚Äî STRUCTURALLY IMPOSSIBLE. The only 101 constructor is downstream of the inline `.await`; there is no detached/spawned path that emits a 101. The pre-fix `run_h1_ws_upgrade_task` (which dialed after 101) is deleted. No race window exists.
  - *Smuggled pipelined request after upgrade headers* ‚Äî `hyper::upgrade::on(&mut req)` is armed only after upstream success (line 1282); on failure it is never armed, `error_response` keeps the wire in H1, smuggled bytes remain in hyper's buffer for normal H1 parsing (subject to the smuggle detector), never blindly forwarded. Proof `no_smuggled_request_forwarded_on_failure` confirms `/smuggled` never reaches the backend.
  - *traceparent verbatim leak* ‚Äî proof asserts parent-id replaced by LB span id, trace-id preserved. Confirmed.
- **Caveat (non-blocking)**: `handshake_headers` (Sec-WebSocket-Accept) is computed from the client `Sec-WebSocket-Key` via `build_handshake_response_headers(&req)`, not echoed from the upstream's 101. RFC 6455 Accept is a pure function of the client key so the value is identical for a conformant upstream; however upstream-negotiated `Sec-WebSocket-Extensions` / selected sub-protocol are NOT reflected from the upstream response (the proxy echoes the client's first offered protocol). This is an extension-negotiation correctness gap, NOT a smuggling/security defect ‚Äî the headline vuln is fully closed.
- **Verdict**: **Verified-Fixed (verifier=verify, 6253ad9a)** ‚Äî non-blocking caveat on WS extension reflection logged for a follow-on.
