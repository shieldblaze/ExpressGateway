# Plan for REL-2-05 — Wire HealthChecker (passive health, required slice)
Finding-ref:     REL-2-05 (high, Open)
Files touched:
- `crates/lb-health/src/lib.rs` (extend the existing `HealthChecker`
  with a per-backend passive-failure tracker; keep the active-probe
  loop intact for the deferred slice)
- `crates/lb-observability/src/lib.rs` (backend-health metrics)
- *Wiring point* — `crates/lb/src/main.rs` belongs to `code` per
  synthesis §D; rel's plan describes the API surface and the metric
  contract. `code`'s CODE-2-13 plan calls into the API rel defines
  here.

Scope (per lead §E.6):
Required-for-prod = **passive health only**. Active health-check loop
(periodic TCP/HTTP probes) remains in `lb-health` but stays unwired
in `main.rs` for this round; it ships in a follow-up. Deferred:
distributed control-plane backends.

Approach:

1. New trait/struct in `lb-health`:
   ```rust
   pub struct PassiveHealth {
       pub listener:        String,
       pub backend:         SocketAddr,
       pub failure_count:   AtomicU32,         // consecutive failures
       pub success_count:   AtomicU32,         // consecutive successes
       pub state:           AtomicU8,          // 0=Healthy, 1=Suspect, 2=Dead
       pub last_transition: AtomicU64,         // unix-millis
   }

   impl PassiveHealth {
       pub fn record_success(&self);
       pub fn record_failure(&self, kind: FailureKind);
       pub fn is_eligible(&self) -> bool;       // Healthy or Suspect
       pub fn cooldown_elapsed(&self, now: Instant) -> bool;
   }
   ```
   State machine:
   - Healthy → Suspect on 1 failure; Suspect → Healthy on 1 success.
   - Suspect → Dead on `failure_threshold` (default 3) failures.
   - Dead → Suspect after `cooldown_ms` (default 30 s); next request
     probes the backend (passive trial).

2. `BalancerExt::pick_with_health(&self, registry: &HealthRegistry)
   -> Result<&Backend, PickError::NoHealthy>` — filters by
   `is_eligible()` before round-robin. `code` calls this in the
   accept path (CODE-2-13).

3. Failure hooks (called from connect/upstream paths owned by code):
   - `connect_timeout`, `connect_refused`, `connect_unreachable` →
     record_failure.
   - upstream 5xx response → record_failure (configurable per route).

4. Metrics (rel-owned):
   - `backend_health_state{listener,backend}` — Gauge with value
     0/1/2 mapped from `state`.
   - `backend_health_transitions_total{listener,backend,to}` — Counter
     incremented on each AtomicU8 store.
   - `backend_eligible{listener}` — Gauge: count of `is_eligible()`
     backends per listener. Alerts can pivot on this dropping to 0.

5. Config (in `[health]` section, new):
   ```toml
   [health]
   passive_failure_threshold = 3
   passive_cooldown_ms       = 30000
   ```

6. ConfigManager and `lb-cp-client` are explicitly **out of scope
   here** — both deferred per lead §E.6. The PassiveHealth surface
   replaces them for round-3 sign-off; CODE-2-13 documents the
   deferred slice.

Proof:
`crates/lb-health/tests/passive.rs`:
1. `test_healthy_to_dead_then_recover`: drive 3 failures → state==Dead;
   sleep `cooldown_ms`; record_success → state==Healthy. Assert
   `backend_health_transitions_total{to="dead"}` = 1,
   `{to="healthy"}` = 2 (initial + recovery).
2. `test_pick_with_health_skips_dead`: two backends; mark one Dead;
   100 picks return only the healthy one.
3. `test_no_eligible_returns_typed_error`: mark both Dead; `pick_with_health`
   returns `PickError::NoHealthy`.

Integration proof in CODE-2-13's plan (not duplicated here): a dead
peer is removed from rotation within `failure_threshold` consecutive
requests, observable via `/metrics`.

Risk / blast radius:
Atomics-only state machine — no locks, no allocation in the hot
path. Worst case: an oscillating backend (Suspect↔Healthy) inflates
the transitions counter; acceptable since transitions are bounded by
request rate. The failure-threshold default (3) may be too eager
for some upstreams; configurable.
