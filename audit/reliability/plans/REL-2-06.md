Approach:
Currently `crates/lb/src/main.rs:931-936` calls:
```rust
tracing_subscriber::fmt().with_env_filter(...).init();
```
which selects the default text formatter. The workspace already
enables `tracing-subscriber/json`, so the switch is a one-liner —
but we centralise it and add structured context.

New `lb-observability::init_tracing()`:
```rust
pub fn init_tracing(cfg: &TracingConfig) -> Result<(), TracingError> {
    let env_filter = EnvFilter::try_from_default_env()
        .unwrap_or_else(|_| EnvFilter::new(&cfg.default_directive));

    let fmt_layer = match cfg.format {
        LogFormat::Json => fmt::layer()
            .json()
            .flatten_event(true)
            .with_current_span(true)
            .with_span_list(false)
            .with_target(true)
            .with_thread_ids(false)
            .with_file(false)
            .with_line_number(false)
            .with_timer(UtcTime::rfc_3339())
            .boxed(),
        LogFormat::Text => fmt::layer()
            .with_target(true)
            .with_timer(UtcTime::rfc_3339())
            .boxed(),
    };

    let registry = Registry::default()
        .with(env_filter)
        .with(fmt_layer);

    // Trace layer for REL-2-07 is added later via .with(otel_layer).
    registry.try_init()?;

    // One-shot startup line with build context.
    tracing::info!(
        version = env!("CARGO_PKG_VERSION"),
        pid = std::process::id(),
        host = %hostname::get().unwrap_or_default().to_string_lossy(),
        "tracing initialised"
    );
    Ok(())
}
```

Config (`[observability.tracing]`, new):
```toml
[observability.tracing]
format            = "json"        # "json" | "text"
default_directive = "info,hyper=warn,quiche=warn"
```
Env-var override: `LB_LOG_FORMAT=text` flips at runtime for dev.

JSON schema fields (will land in `METRICS.md` / `RUNBOOK.md` via
REL-2-01):
- `timestamp` (RFC 3339 UTC)
- `level` (info/warn/error/debug)
- `target` (rust module path)
- `fields` (flattened message + structured fields)
- `span` (current span name + id + parent)
- For request spans (REL-2-07): `trace_id`, `span_id`,
  `http.method`, `http.target`, `http.status_code`, `route`,
  `listener`, `backend`.

Proof:
`tests/log_format.rs`:
1. `test_json_format_default`:
   spawn binary with `format = "json"`; capture stdout line 1; parse
   with `serde_json::from_str`; assert keys `timestamp`, `level`,
   `target`, `fields.version` present. Assert
   `fields.version == env!("CARGO_PKG_VERSION")`.
2. `test_text_format_env_override`:
   spawn with `LB_LOG_FORMAT=text`; assert stdout line 1 starts
   with an RFC-3339 timestamp and the message is human-readable
   (parser refuses JSON).
3. `test_request_log_carries_span_context`:
   send 1 H1 request; find the corresponding info-level log line;
   assert `trace_id` and `span_id` present (gated on REL-2-07
   landing first; until then, just `span.name == "lb.request"`).

Risk / blast radius:
Output-format change only; no behaviour change in the hot path. Risk
of breaking operators who already grep text logs — mitigated by the
text env-override and a one-cycle deprecation note in `RUNBOOK.md`.
Volume note: JSON is ~30% larger per line; log shipper capacity needs
re-checking (called out in the doc sweep).
