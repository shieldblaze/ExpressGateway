# Plan for SEC-2-10 — TLS-handshake timeout on `acceptor.accept()`

Finding-ref:     SEC-2-10 (medium, current status: Open)

Files touched:
- `crates/lb-security/src/lib.rs` (re-export `timeout_accept`)
- `crates/lb-security/src/tls_accept.rs` (new module; sec owns)
- `crates/lb-security/tests/tls_accept_timeout.rs` (new)
- (consumer edit lives in `crates/lb/src/main.rs` — code-owned per
  §D; sec's plan only provides the helper)

Joint boundary with code:
- §D assigns `crates/lb/src/main.rs` to code. The TLS-accept call
  site at the listener loop is touched by code's CODE-2-01 +
  CODE-2-03 + CODE-2-05 plans. **Sec provides the
  `timeout_accept` helper; code's CODE-2-01 plan inserts the call**
  in the same accept-loop rewrite. The exact API:
  ```
  pub async fn timeout_accept<IO>(
      acceptor: &TlsAcceptor,
      stream: IO,
      timeout: Duration,
  ) -> Result<TlsStream<IO>, TlsAcceptError>
  where IO: AsyncRead + AsyncWrite + Unpin;
  ```
  Returns `Err(TlsAcceptError::Timeout)` on elapsed and
  `Err(TlsAcceptError::Handshake(e))` on rustls error. Code's call
  site logs the IP and drops the connection.

Approach:
Implement `timeout_accept` as a thin `tokio::time::timeout` wrapper
around `acceptor.accept(stream)`. Default timeout: 5 seconds
(configurable via `[security].tls_handshake_timeout_ms`, also
exposed via `AcceptOptions::tls_handshake_timeout`).

The function additionally records two metrics (counters):
- `tls_handshake_timeouts_total{listener_name}` — every elapsed
- `tls_handshake_errors_total{listener_name,error_class}` — every
  rustls error (cert error / version error / etc.)

`error_class` is derived from the rustls `Error` variant; this lets
the operator distinguish slowloris (timeout) from legitimate-client
mis-config (handshake error) — different remediation.

Approach to integration with SEC-2-04 (per-IP cap): the per-IP
permit is acquired **before** the TLS accept, then held across the
handshake. If the handshake times out, the permit is released by
RAII — so timeout-floods cannot circumvent the per-IP cap.

Proof:
- `cargo test -p lb-security --test tls_accept_timeout::test_byte_per_minute_attacker_rejected`
  — start a test listener wired through `timeout_accept(5s)`, open
  a raw TCP socket, send 1 byte of ClientHello, sleep 6s; assert
  `TlsAcceptError::Timeout` is returned within 6s wall-clock; assert
  the connection is closed and `tls_handshake_timeouts_total`
  counter incremented by 1.
- `cargo test -p lb-security --test tls_accept_timeout::test_normal_handshake_completes`
  — drive a tokio-rustls client through a full TLS 1.3 handshake;
  assert `Ok(TlsStream)` within 100ms.
- `cargo test -p lb-security --test tls_accept_timeout::test_handshake_error_distinguished`
  — send a bogus ClientHello (wrong protocol version); assert
  `Err(TlsAcceptError::Handshake)` not `Err(::Timeout)`; assert
  the error counter labels distinguish.
- Invariant: elapsed time on the timeout-path is ≤ `timeout +
  100ms` (tokio scheduler slop).

Risk / blast radius:
- 5s default may reject legitimate clients on extremely lossy links
  (≤2G mobile, satellite). Mitigation: per-listener override;
  documented in RUNBOOK. Real-world: TLS 1.3 handshakes complete in
  <1s on >99.9% of connections.
- Boundary risk in main.rs: code must reference `timeout_accept` in
  CODE-2-01's call-site insertion. Sec's API name and signature are
  finalised in this plan; code's plan must use them verbatim.
- No source-conflict — sec's edits land in `lb-security` (new file).
