Approach:
Two artefacts. First, an attack-evidence note
(`SEC-2-02-evidence.md`) showing one attacker with 64k src-ports per
IP saturates the 1M IPv4 map in <16 IPs; userspace TTL sweep cannot
keep up (1M-entry delete scan ≥30s; flood refills in <1s).
Conclusion: **kernel LRU is the only viable design**.

Second, a smoke test `tests/security_conntrack_lru_flood.rs` gated
on `#[cfg(feature = "xdp-online")]`:
1. Attach eBPF to a veth pair.
2. Install one legitimate sticky flow; drive 1 pkt/s on it.
3. From another netns, flood unique random 5-tuples at
   ≥2×map_size pps for 30s; observe `xdp_conntrack_entries{v=4}`
   reach cap.
4. Continue the legitimate flow 30s more.
5. Assert: legitimate flow's entry still present (LRU promoted on
   each lookup). Fails if evicted.

This test catches the regression where ebpf accidentally drops the
LRU lookup-bump (kernel LRU only promotes on lookup, not insert).

Proof:
- `cargo test -p lb-l4-xdp --test security_conntrack_lru_flood
  --features xdp-online -- --ignored test_legitimate_flow_survives_flood`
  (ignored by default; CI runs it nightly on a kernel-equipped runner
  per ebpf's EBPF-2-07 CI plan).
- Invariants:
  - Legitimate flow's CONNTRACK entry is still present after the
    flood (assert `bpftool map lookup` returns the original
    backend ID).
  - Eviction count metric > map_size during flood (proves LRU is
    evicting, not refusing).
  - Attacker 5-tuples eventually disappear (FIFO-ish recency).
- Static evidence: `SEC-2-02-evidence.md` records the rate-math and
  the conclusion that no userspace evictor can keep up.

Risk / blast radius:
- Smoke test needs CAP_BPF + kernel ≥5.8 + veth; runs only on ebpf's
  CI matrix (EBPF-2-07). Local `cargo test` skips it.
- No source-conflict with ebpf in Round 4 (sec only touches
  `tests/` + evidence note).
- If aya `LruHashMap` lacks an accessor we need, fallback is a
  100ms-tick userspace TTL evictor — but that voids the smoke
  test's guarantee, so sec must downgrade fix status before
  accepting fallback.
