# Plan for SEC-2-03 — Wire SlowlorisDetector / SlowPostDetector

Finding-ref:     SEC-2-03 (medium, current status: Open)

Files touched:
- `crates/lb-l7/Cargo.toml` (lb-security dep — same edit as SEC-2-01;
  one Round-4 edit covers both plans)
- `crates/lb-l7/src/h1_proxy.rs` (request-header + request-body wrap)
- `crates/lb-l7/src/h2_proxy.rs` (DATA-frame pacing on request side)
- `crates/lb-security/src/slowloris.rs` (add a builder API that
  wraps an `AsyncRead`; sec owns)
- `crates/lb-security/src/slow_post.rs` (same)
- `crates/lb-l7/tests/slowloris_wired.rs` (new)
- `crates/lb-l7/tests/slow_post_wired.rs` (new)

Joint boundary:
- `proto` owns `h2_proxy.rs` per §D. Sec adds the API
  (`SlowPostDetector::observe_chunk(len, now) -> Result<(),
  SecurityError>`); proto inserts one call per H2 DATA frame
  alongside PROTO-2-10 / SEC-2-01 inserts.
- TLS-accept timeout is handled by SEC-2-10 (separate plan); this
  plan does **not** touch the accept site.

Approach:
Extend `SlowlorisDetector` with a builder that produces a wrapper
around the header-read future:
```
SlowlorisDetector::new(min_bytes_per_tick, tick = 1s)
    .wrap_header_read(stream)
    -> impl Future<Output = Result<HeadersDone, SecurityError>>
```
The wrapper polls the inner future and on each `Pending` checks
elapsed wall-clock and bytes-read-so-far via the new
`HeaderProgress` trait sec adds.

Extend `SlowPostDetector` with `observe_chunk(len, now)` returning
`Err(SecurityError::SlowPost)` when the running rate drops below
`min_body_bytes_per_second`. Defaults: 256 B/s; configurable via
`[security].slow_post_min_bps`.

Wire in `H1Proxy::handle`:
- Wrap the hyper request header read with `SlowlorisDetector` (5 s
  total cap, 64 B/s min).
- Pipe the body stream through `SlowPostDetector::observe_chunk` on
  every `Frame::Data`.

Wire in `H2Proxy::handle` (proto inserts the call; sec provides the
API). Per-DATA-frame `observe_chunk`. H2 has no slowloris analogue
(headers arrive in one HEADERS frame), so SlowlorisDetector is H1
only.

Defaults are deliberately permissive (256 B/s) to avoid breaking
mobile / lossy-network legitimate clients. Operators can tighten.

Proof:
- `cargo test -p lb-l7 --test slowloris_wired::test_slow_header_byte_per_30s_rejected`
  — drives a TCP stream sending 1 byte / 30 s; asserts connection
  closed within 5 s with `SecurityError::Slowloris`.
- `cargo test -p lb-l7 --test slowloris_wired::test_normal_client_unaffected`
  — drives a normal hyper client with realistic timing; asserts no
  rejection (regression guard against tight defaults).
- `cargo test -p lb-l7 --test slow_post_wired::test_slow_body_rejected`
  — POST body sending 1 B every 10 s; asserts 408 returned within
  20 s.
- `cargo test -p lb-l7 --test slow_post_wired::test_slow_body_below_threshold_passes`
  — POST at exactly threshold rate (regression guard).
- Invariant: detector errors propagate as 408 (Request Timeout) per
  RFC 9110 §15.5.9, not 400 (smuggling rejection).

Risk / blast radius:
- False positives on slow mobile uploads. Mitigation: per-listener
  override of `min_body_bps` to 0 to disable. Default 256 B/s is
  below all known legitimate clients (≥1 kB/s typical even on 2G).
- Adds one `Instant::now()` per body frame — measurable but
  sub-µs. Bench impact <1%.
- Boundary risk with rel F-05 (TLS accept slowloris): SEC-2-10
  handles that separately; this plan does NOT wrap acceptor.accept.
