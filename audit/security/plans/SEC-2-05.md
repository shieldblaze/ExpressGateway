# Plan for SEC-2-05 — 0-RTT replay window resizing under spray

Finding-ref:     SEC-2-05 (medium, current status: Open)

Files touched:
- `crates/lb-security/src/zero_rtt.rs` (replace fixed HashSet)
- `crates/lb-security/tests/zero_rtt_spray.rs` (new)
- `crates/lb-security/Cargo.toml` (add `growable-bloom-filter` or
  build a sharded ring; see Approach)

Joint boundary:
- Sole-owned by sec. No other plan touches `zero_rtt.rs`. The
  consumer is `lb-quic`'s 0-RTT acceptance path (code-owned crate)
  but the API surface does not change — `ZeroRttReplayGuard::observe`
  returns the same `Result<bool, _>`. So no joint boundary.
- One soft-link: SEC-2-13 confirmed 0-RTT is **not** enabled on
  TCP/TLS (rustls `max_early_data_size = 0` by construction). So this
  plan only matters for QUIC 0-RTT.

Approach:
Replace the fixed `HashSet<Token>` with a **time-windowed
sharded-bloom-filter**. Design:

1. `ZeroRttReplayGuard` holds two bloom filters (`current`,
   `previous`) and an `Instant` rotation deadline. Rotation period =
   `max_age` from `RetryTokenSigner` (configured at construction).
2. On `observe(token)`:
   - If `now > deadline`, swap `previous = current; current =
     Bloom::new(); deadline = now + max_age`.
   - Test `previous.contains(token) || current.contains(token)`.
   - Insert into `current`.
3. Bloom params: target false-positive rate 1e-6 at peak-RPS ×
   max_age. Default peak-RPS = 100_000; max_age = 10s → 1M
   insertions per window → ~3 MB per filter × 2 windows = 6 MB
   constant memory.

The two-window scheme guarantees a token is detected as replay
for at least `max_age` seconds after first seen (matching
`RetryTokenSigner::max_age`), which is the security boundary —
beyond that the retry signature is rejected anyway.

Add counters:
- `zero_rtt_replays_total` (incremented on detection)
- `zero_rtt_evictions_total` (per rotation: the count from `previous`
  before drop)
- `zero_rtt_bloom_fpp_estimate` (gauge — current load factor)

Refuse 0-RTT (return `Err(SecurityError::ZeroRttOverload)`) when
the *current* window's insertion rate exceeds 2× expected peak — a
hard signal of spray attack. Operators can disable via
`[security].zero_rtt_under_attack_action = "accept"`.

Proof:
- `cargo test -p lb-security --test zero_rtt_spray::test_replay_detected_after_1m_unique_tokens`
  — insert 1_000_000 unique tokens, then a captured token, then
  replay the captured token; assert replay detected.
- `cargo test -p lb-security --test zero_rtt_spray::test_replay_detected_across_window_rotation`
  — insert token, wait > max_age/2, rotate, replay token; assert
  detected (previous window still active).
- `cargo test -p lb-security --test zero_rtt_spray::test_replay_window_after_2x_max_age`
  — insert token, wait 2 × max_age (two rotations), replay; assert
  NOT detected (intentional — RetryTokenSigner expires first).
- `cargo test -p lb-security --test zero_rtt_spray::test_bloom_fpp_under_load`
  — insert peak-load tokens, run 10_000 negative queries, assert
  false-positive count ≤ 100 (1e-2 worst-case under load).
- Invariant: memory usage is bounded at `2 × bloom_size_bytes`
  regardless of spray rate.

Risk / blast radius:
- Bloom filter false-positives reject legitimate 0-RTT replays.
  Acceptable: a legitimate replay is rare (resend after timeout) and
  a 1-in-1M false rejection cleanly degrades to 1-RTT.
- Increased memory (6 MB constant) — well under budget.
- The 2× rate-spike refusal heuristic could trigger on legitimate
  traffic spikes. Mitigation: configurable threshold; default to
  accept-with-warning rather than refuse.
- No other team touches `zero_rtt.rs`. Zero edit conflict.
